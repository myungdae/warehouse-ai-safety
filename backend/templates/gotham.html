🏗️ Palantir 방식: Design-time vs Runtime 완전 분리
🏗️ Design-time (CI/CD 배포 전)
• Object Type Registry — Track, Drone, Zone, Event 10개 타입 정의
• RDF/OWL (.ttl) — 클래스·속성·제약 명세 (시맨틱)
• SHACL 제약 — 스키마 무결성 검증, 배포 전 실행
• Link Type Definition — DETECTS, TRIGGERS 6개 관계
→ 온톨로지스트 + 도메인 전문가 설계.
런타임에는 참조 X — 스키마만 존재
⚡ Runtime (작전 중 실시간)
• Neo4j Property Graph — Object Instance 저장·쿼리
• Redis Streams — 센서 데이터 실시간 PubSub
• Event Engine (rules.py) — Python 룸열 실행
• Cypher 쿼리 — 그래프 탐색, 상황 질의
→ RDF/OWL/SPARQL 미사용.

🏗️ 아키텍처 통합 위치 — L2.5 Expert Knowledge Layer
L1 센서 → L2 Ontology → L2.5 ✏️ Expert KB (MOE) → L3 Event Engine → L4 Situation → L5 Intent → L6 RAG+LLM ← 전문가 지식 주입 → L7 Action
MOE(L2.5)는 Operational Ontology(L2)와 Event Engine(L3) 사이에 위치하여, 전문가의 도메인 지식이 시나리오 KB와 함께 L6 LLM에 컨텍스트로 주입됩니다.
질의 시 RAG가 Scenario KB(E001-RULE012)와 Expert KB(EXP001-) 를 통합 검색하여 출처를 구분해 응답합니다.
단계 행위자 입력 처리 출력
1️⃣ 지식 입력 군사 전문가 아티클·교범·경험 MOE 입력창 localExpertKB[]
2️⃣ 색인 생성 RAG Engine 본문+태그 텍스트 TF-IDF tokenize 벡터 색인 즉시 반영
3️⃣ 질의 처리 지휘관 (자연어) "NLL 드론 교전 기준은?" RAG.query() Top-3 Scenario+Expert 통합 검색
4️⃣ 응답 생성 LLM (L6) RAG 컨텍스트 주입 GPT-4o 응답 생성 [EXP001·전문가] 배지 표시
✓ 작업이 완료되었습니다.
